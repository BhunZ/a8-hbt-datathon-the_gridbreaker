# `order_items.unit_price` — Formula Derivation

**Status:** Empirically verified on all 714,669 order-item rows.
**Takeaway:** `unit_price` is **not** a deterministic function of `products.price` alone. It embeds a **promo discount** *and* a **Gaussian noise term**. That is why `price ± shipping_fee` and `price − discount` never reconcile.

---

## 1. The Formula

```
unit_price  =  products.price  ·  (1 − d)  ·  (1 + ε)

where
    d = promotions.discount_value / 100       if order_items.promo_id is not null
        0                                      otherwise
    ε ~ Normal(0, σ ≈ 0.020)                  per-line, i.i.d. (2% std noise)
```

Both `promo_type = 'fixed'` and `promo_type = 'percentage'` interpret `discount_value` as a **percent**. There is **no** flat-dollar promo in the data. The only difference is the set of values used:

| promo_type   | discount_value values observed |
|--------------|--------------------------------|
| `fixed`      | `{50}`                          |
| `percentage` | `{10, 12, 15, 18, 20}`          |

## 2. Empirical Validation

Residual `ε = unit_price / (list_price · (1 − d)) − 1`, stratified by promo bucket (output from `unit_price_model.py`):

| Bucket                    | n       | mean ε   | std ε   | P(|ε| < 3σ) |
|---------------------------|--------:|---------:|--------:|-------------:|
| NO_PROMO                  | 438,353 | −0.00001 | 0.02000 | 0.9973       |
| fixed / dv = 50           |  20,950 | −0.00010 | 0.01998 | 0.9971       |
| percentage / dv = 10      |  45,654 | −0.00005 | 0.01988 | 0.9968       |
| percentage / dv = 12      |  71,591 | −0.00004 | 0.01993 | 0.9975       |
| percentage / dv = 15      |   8,905 | −0.00024 | 0.02005 | 0.9979       |
| percentage / dv = 18      |  57,417 | +0.00014 | 0.02001 | 0.9975       |
| percentage / dv = 20      |  71,799 | −0.00003 | 0.02006 | 0.9974       |

Every bucket has the **same** noise σ ≈ 0.0200 and the **same** Gaussian tail (~99.7% inside 3σ). That uniformity is the signature of an i.i.d. multiplicative noise layer added *after* the list-price / discount computation — almost certainly a simulation of real-world intra-day price dispersion (anchoring, A/B pricing, regional variance).

## 3. What `discount_amount` Actually Represents

```
discount_amount  =  quantity  ·  unit_price  ·  d
```

Note the baseline: `discount_amount` is computed from the **already-charged** `unit_price`, *not* from `products.price`. That is why the team's naive attempt of `unit_price = list_price − discount_amount` fails — the formula is circular.

**Evidence:** of the 276,316 promo rows, **252,989 (91.6%)** match exactly. The remaining ~8.4%:
- 23,121 single-promo rows miss by **±$0.01** (half-cent rounding — not a real mismatch).
- 206 stacked-promo rows (`promo_id_2` populated) have an additional **flat +$50** component tied to the second promo.

## 4. Why the Team's Two Hypotheses Failed

| Attempted formula                      | Why it is wrong |
|----------------------------------------|-----------------|
| `products.price + shipping_fee`        | `shipping_fee` is **order-level** (in `shipments`), never added into `unit_price`. Shipping is reconciled separately at the invoice (`payments.payment_value`). |
| `products.price − discount_amount`     | `discount_amount` is already derived *from* `unit_price`; using it to reconstruct `unit_price` is circular. Also, it does not account for the 2% noise term. |

## 5. Corollaries Useful for Presentation

1. **`sales.Revenue = Σ (quantity × unit_price)`** across **all** order-items (every status). No filtering, no adjustments. MAE = 0.00.
2. **Margin** at the line level is computed **after** the noise and promo transformation: `margin_line = quantity · (unit_price − products.cogs)`. Do **not** use list price.
3. **Invoice reconciliation** (for the payments table): `payment_value ≈ Σ(qty · unit_price) − Σ(discount_amount) + shipping_fee`, with median residual ≈ $1.50 per order (rounding + the flat-$50 stacked component).
4. **"Saved" dollars visible to the customer** are `discount_amount`. The *true* economic savings vs MSRP are `quantity · (list_price − unit_price)`, which is strictly larger (because `discount_amount` uses the post-discount base).

## 6. Recommended Slide Language (for the presentation deck)

> *"Line-level charged prices are generated by taking the product's list price, applying the promotional discount percentage when one is attached, and then perturbing the result with ~2% per-line Gaussian noise. The `discount_amount` column records the customer-visible savings computed off the charged price, not off the MSRP. Because of the stochastic term, `unit_price` cannot be re-derived exactly from any combination of `products.price`, `discount_value`, `discount_amount` and `shipping_fee` — it can only be predicted up to a 2% residual."*
